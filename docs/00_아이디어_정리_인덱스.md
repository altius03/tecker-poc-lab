## 공통 선정 기준

- 문제와 대상 사용자가 명확한가
- 4주 안에 MVP 구현이 가능한가
- 프론트/백엔드/AI/인프라 역할 분리가 가능한가
- 적당한 재미 요소가 있는가
- 상업성과 돈 낼 주체가 명확한가
- 확장 가능성이 있는가
- 발표에서 데모로 보여주기 쉬운가

## 공통 해결 원칙

| 원칙 | 의미 |
| --- | --- |
| 기능 보존이 기본 | 어려운 기능도 유지하고, 권한·문구·검증 단계·provider mode로 구현 가능 경로를 만든다. |
| PoC는 핵심 병목 검증 | 예쁜 화면보다 “이 서비스가 성립하려면 반드시 풀어야 하는 문제”를 실제 해결 흐름으로 보여준다. |
| 법/정책/과금 리스크를 명시 | 위험 도메인, 데이터 수집, 지도/리뷰 정책, API 과금은 피하지 않고 MVP 대응 방식을 적는다. |
| 발표는 1~2분 흐름 | 너무 거창한 아키텍처 설명보다 입력 → 처리 → 결과 → 해결한 병목을 보여준다. |

## 세 아이디어 비교

| 주제 | 핵심 문제 | 핵심 사용자 | 핵심 PoC | 주요 리스크 | 해결 방향 |
| --- | --- | --- | --- | --- | --- |
| VOC Quest | 공개 시장 VOC가 여러 채널에 흩어져 있고, 제품명 별칭/비꼼/노이즈 때문에 담당자가 놓치기 쉬움 | 브랜드 마케팅, PM, CS, 커머스 브랜드, 대행사 | 제품 seed → 공개 VOC 자동 수집 → 보유 VOC 보강 → 정제/클러스터링 → 개선 퀘스트 | 합법적 수집, API 쿼터, 별칭, 비꼼, 개인정보, 실행 가능한 퀘스트 품질 | 공식 API/공개 페이지 리더/고객 제공 VOC를 권한 라우팅으로 연결 |
| Be:Careful | 가족 돌봄 정보가 처방전, 약봉투, 카톡, 기억 속에 흩어져 복약/일정 공백이 생김 | 가족 보호자, 방문요양센터, 실버케어 기관 | OCR 텍스트 → 약명/복용 일정 후보 → 보호자 확인 → DUR 주의 → 인수인계 카드 | 의료 판단 오인, 민감정보, OCR 오류, 알림 비용, 커머스 리스크 | 복용 지시가 아니라 확인 필요 정보와 전문가 상담 질문으로 제공 |
| 데이트 동선 탐색 | 지도/블로그/SNS를 오가며 예산, 무드, 이동수단, 동선을 수동 조립해야 함 | 데이트 장소·동선 고민이 잦은 커플 | 자연어 조건 → 구조화 → 장소 후보 → 코스 순서 → 근거 → 길찾기 | 리뷰 데이터 확보, 지도 provider 정책, Google Places 과금, 단순 추천화 | provider mode로 Google/Kakao 데이터와 지도 표시를 맞춤 |
