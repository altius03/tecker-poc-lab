## 1. 서비스 개요

### 1-1. 프로덕트 제목

데이트 동선 탐색 프로그램

### 1-2. 한 줄 소개

자연어로 데이트 조건을 입력하면, 지도 위에서 장소를 검색·필터링하고 **여러 곳을 잇는 데이트 코스**까지 설계해 길 찾기로 연결해 주는 서비스. (하나의 장소가 아닌 코스가 메인 테마)

### 1-3. 프로젝트 목표 / 핵심 컨셉

이 프로젝트는 **추천 알고리즘**이 아니라 **"자연어 조건 해석 → 공간 필터링 → 동선 설계"**라는 **명시적·설명 가능한 파이프라인**이다.

---

## 2. 문제 정의

- 커플의 반복적인 고민 1순위는 **"오늘 어디서 뭐 하지?"** 이고, 실제로는 **여러 장소를 순서대로 엮는 동선 짜기**가 더 어렵다.
- 기존 지도 앱의 단순 필터(카테고리·별점순 정렬)는 **복합 조건**("○○역 근처, 예산 10만 원, 분위기 좋은 카페랑 파스타집, 걸어서 이동")을 반영하지 못한다.
- 블로그·SNS의 "데이트 코스 추천" 글은 **내 위치·예산·시간에 맞춤이 아니고**, 지도/길찾기로 바로 이어지지도 않는다.

**결과:** 앱 여러 개를 옮겨 다니며(검색앱 → 지도앱 → 길찾기앱) 수동으로 동선을 조립하는 비효율이 발생한다.

## 4. 주요 기능

### 4-1. 자연어 조건 해석

- LLM으로 입력을 구조화 JSON으로 추출(지역·반경·예산·무드·카테고리 세트·이동수단·시간대).

### 4-2. 조건별 데이터 검증 가능성 분리

- 사용자의 조건을 `verifiable`, `proxy_verified`, `proxy_not_found`, `unsupported`로 분류한다.
- 지역, 카테고리, 평점, 리뷰 수, 영업시간, 좌표/거리처럼 Google Places 필드나 계산으로 확인 가능한 조건은 `verifiable`로 둔다.
- 조용한, 분위기 좋은, 데이트하기 좋은 같은 무드 조건은 리뷰 근거 문장이 있을 때만 `proxy_verified`로 둔다.
- 정확한 예산, 실시간 웨이팅, 예약 가능 여부처럼 안정적인 공식 필드가 없는 조건은 `unsupported`로 분리하고 결과 메시지에 명시한다.

### 4-3. 지도 기반 장소 검색

- 지역·카테고리별 장소 후보 조회.

### 4-4. 데이터 부족 조건 처리

- 검증 가능한 조건만 추천 기준에 사용한다.
- 검증하지 못한 조건은 숨기지 않고 `unmatched_conditions`에 이유와 함께 남긴다.
- 예: `예산 12만 원 이하`는 메뉴 가격 데이터가 없어 직접 검증하지 못했다고 표시한다.

### 4-5. 후보 0개 방지

- 조건을 전부 강하게 적용하지 않고 우선순위를 둔다.
- 지역, 카테고리, 이동수단/거리처럼 코스 성립 조건은 유지하고, 무드·평점·리뷰 수·반경은 후보 수에 따라 완화한다.
- 완화 과정은 `fallback_log`에 기록한다.

### 4-6. 동선 설계

- 선택된 장소들을 잇는 **경유지 경로**를 생성하고 이동수단·소요시간을 계산.

### 4-7. 길찾기 연결

- 완성된 코스를 Google Maps 길찾기 URL로 연결한다.
- 리뷰 근거와 장소 후보는 Google Places 기준으로 통일하고, 경로 최적화 API는 이번 PoC 범위에서 제외한다.

### 4-8. 리뷰 근거 추출

- 최종 후보 일부의 Google Places 리뷰를 가져와 LLM으로 무드 조건의 근거 문장을 추출한다.
- 근거 문장이 없으면 해당 무드 조건은 `proxy_not_found`로 표시한다.

## 5. 서비스 흐름

### 5-1. 1~2분 발표 흐름

1. 사용자가 지역, 예산, 무드, 카테고리, 이동수단을 자연어로 입력한다.
2. LLM이 입력 문장을 구조화 조건 JSON으로 변환한다.
3. rule table이 조건을 `verifiable`, `proxy`, `unsupported`로 분류한다.
4. Google Places로 장소 후보를 수집하고, 최종 후보 리뷰에서 무드 근거 문장을 추출한다.
5. 검증하지 못한 조건은 숨기지 않고 결과 메시지에 표시한다.
6. 후보가 부족하면 낮은 우선순위 조건부터 완화하고 `fallback_log`를 남긴다.
7. 후보 장소를 역할 순서로 간단히 연결하고 Google Maps 길찾기 URL을 생성한다.

## 6. 챌린징 포인트 / PoC

### 6-1. 검증할 챌린징 포인트

| 챌린징 포인트 | 확인할 내용 | PoC 해결 방식 |
| --- | --- | --- |
| 조건별 데이터 검증 가능성 분리 | 사용자 조건 중 무엇을 실제 데이터로 검증할 수 있고 무엇을 검증할 수 없는지 확인 | LLM 조건 JSON + rule table로 `verifiable/proxy/unsupported` 분류 |
| 데이터 부족 조건 처리 | 정확한 예산, 웨이팅, 예약 가능 여부처럼 데이터가 부족한 조건을 어떻게 처리할지 확인 | 추천 기준에서 제외하되 `unmatched_conditions`에 이유를 명시 |
| 무드 조건 근거 확인 | 조용한/분위기 좋은/데이트하기 좋은 조건이 리뷰에서 근거 문장으로 확인되는지 확인 | 최종 후보 일부의 Google Places 리뷰를 LLM으로 근거 문장 추출 |
| 후보 0개 방지 | 조건이 많아 후보가 없어지는 상황을 어떻게 복구할지 확인 | unsupported 제외, 무드 조건 근거 전환, 평점/리뷰 수 완화, 반경 확대 순서로 fallback |
| 검색 결과 느슨한 매칭 | Google Text Search가 조건과 느슨한 후보를 많이 반환하는 문제 확인 | 다음 보완 과제로 관련성 점수화 필요 |

### 6-2. 실제 소규모 검증 결과

2026-07-03에 `GOOGLE_MAPS_API_KEY`, `OPENAI_API_KEY`를 사용해 대표 입력 2개를 실행했다.

```text
입력 1: 성수역 근처에서 조용한 파스타집이랑 디저트 카페, 예산 12만 원 안쪽, 도보 이동 가능한 코스로 짜줘
입력 2: 성수역 근처에서 조용한 천문대 레스토랑이랑 사막 테마 디저트 카페, 예산 3만 원 안쪽, 도보 이동 가능한 코스로 짜줘
```

공통 실행 결과:

```text
status: success
candidate_count: 10
condition_capability: verifiable 6, unsupported 1, proxy_not_found 1
review_place_count: 2
review_evidence_count: 0
route_stop_count: 2
```

확인된 점:

- 정확한 예산은 메뉴 가격 데이터가 없어 `unsupported`로 분리됐다.
- 조용한 조건은 최종 후보 리뷰에서 근거 문장을 찾지 못해 `proxy_not_found`로 남았다.
- 후보는 충분히 나와 fallback은 발동하지 않았다.
- Google Text Search는 어려운 조건에도 후보를 넓게 반환하므로, 다음 병목은 후보 0개보다 검색 결과의 느슨한 매칭을 점수화하는 것이다.

## 7. 차별점 / 어필 포인트

| 구분 | 일반 지도 앱 필터 | 본 프로젝트 |
| --- | --- | --- |
| 입력 | 카테고리 + 정렬 옵션 | **자연어 복합 조건**(지역·예산·무드·시간·이동수단) |
| 출력 | 장소 목록 | **한 곳 + 여러 곳을 잇는 코스(동선)** |
| 근거 | 별점순(불투명) | **각 장소가 뽑힌 조건 근거를 명시**(거리·평점·키워드 매칭) |
| 연결 | 개별 길찾기 | **코스 전체를 경유지 길찾기로 한 번에 연결** |

### 7-1. 차별화 포인트 3가지

1. **동선 설계:** 단일 장소가 아니라 **다중 경유 경로**를 만든다. → 기술적 난도의 핵심이자 다른 팀과의 차별점.
2. **맥락 기반 자연어:** "분위기 있는", "조용한", "야경 보이는" 같은 **무드 키워드**를 구조화 조건으로 변환.
3. **근거 투명성:** "왜 이 장소가 코스에 들어갔는지"(3km 이내 · 평점 4.6 · '분위기' 리뷰 다수)를 사용자에게 그대로 보여준다.

## 8. 기술 챌린지 / 기술적 가능성

### 8-1. 위치 기반 검색 / 공간 인덱싱

사용자의 위치나 선택한 지역을 기준으로 주변 후보 장소를 빠르게 찾습니다. 먼저 대략적인 범위를 좁힌 뒤, 실제 거리 계산으로 정확도를 높입니다.

### 8-2. 다중 지점 동선 설계

여러 장소를 방문할 때 어떤 순서로 이동하면 효율적인지 계산합니다. 이동수단별 예상 소요시간을 반영해 현실적인 코스를 구성합니다.

### 8-3. 자연어 조건 구조화

“조용한 분위기”, “데이트하기 좋은 곳”, “비 오면 실내 위주” 같은 표현을 검색 가능한 조건으로 변환합니다. 이를 위해 프롬프트와 응답 스키마를 안정적으로 설계합니다.

### 8-4. 지도 API 연동 및 호출 최적화

지도 API를 활용해 장소, 거리, 이동시간 정보를 가져옵니다. API 사용량을 줄이기 위해 결과를 캐싱하고, 필요한 요청만 묶어서 처리합니다.

## API 호출 과금 검토

| API | 과금/쿼터 판단 | MVP 대응 |
| --- | --- | --- |
| Google Places Text Search | 과금 대상. 장소 후보 검색에 사용 | 대표 입력 1개당 카테고리별 제한 호출 |
| Google Places Place Details / reviews | 리뷰 필드는 비용 리스크가 큼 | 최종 후보 2~3개에만 제한 호출 |
| OpenAI API | 조건 구조화와 리뷰 근거 추출에 토큰 과금 발생 | 짧은 JSON 출력, 최종 후보 리뷰만 분석 |
| Google Routes API | pay-as-you-go. Compute Routes는 요청당, Matrix는 element 단위 과금 | 이번 PoC에서는 사용하지 않고 Google Maps URL 사용 |
| Google Maps URLs | API key 없이 사용 가능 | 경유지 길찾기 데모는 URL 링크로 처리 |
