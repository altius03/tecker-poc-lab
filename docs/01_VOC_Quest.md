## 1. 서비스 개요

### 1-1. 프로덕트 제목

**VOC Quest** : AI 기반 브랜드 시장 VOC 수집·이슈 분석·개선 관리 플랫폼

### 1-2. 한 줄 소개

브랜드/제품명을 입력하면 공개 웹·검색·댓글·커뮤니티 반응을 자동 조사하고, 고객 불만을 개선 퀘스트로 전환하는 브랜드 이슈 관리 서비스

### 1-3. 프로젝트 목표

브랜드가 직접 VOC 파일을 준비하지 않아도 공개적으로 접근 가능한 시장 반응을 자동 탐색한다. 다만 기존 CSV/사내 CS 로그 업로드 기능은 없애지 않고, 자동 수집 결과를 보강하는 내부 VOC 입력 채널로 유지한다.

---

## 2. 문제 정의

> _**"고객 불만은 이미 시장에 퍼져 있지만, 브랜드는 늦게 발견합니다."**_

고객의 불만은 고객센터에만 남지 않습니다. 블로그 후기, 네이버 카페글, 유튜브 댓글, 커뮤니티, 뉴스 댓글, 쇼핑 검색 결과, SNS 반응처럼 여러 공개 채널에 흩어져 있습니다.

하지만 담당자가 매번 제품명과 별칭을 바꿔가며 검색하고, 본문을 열어 읽고, 광고성 글과 무관한 글을 걷어내고, 반복되는 불만을 정리하는 일은 시간이 오래 걸립니다.

특히 사용자는 제품명을 정확히 부르지 않습니다. “파란 앰플”, “이번 신상”, “그 펌프 선크림”처럼 별칭으로 말하거나, “배송 진짜 빠르네요 ㅋ 2주 걸림”처럼 비꼼으로 불만을 표현합니다.

결국 브랜드는 시장에 이미 드러난 문제를 늦게 발견하고, 제품 개선과 CS 대응 타이밍을 놓칠 수 있습니다.

## 3. 타깃 사용자

- 출시 제품에 대한 시장 반응을 빠르게 모니터링해야 하는 브랜드 마케팅 담당자
- 고객 불만을 제품 개선 과제로 전환해야 하는 PM/기획자
- 반복 문의와 부정 여론을 줄이고 싶은 CS 담당자
- 리뷰·블로그·카페·댓글 반응을 정리해야 하는 커머스 브랜드
- 여러 브랜드의 시장 반응을 대리 조사하는 마케팅 대행사

## 4. 주요 기능

### 4-1. 브랜드/제품 seed 입력

- 담당자가 브랜드명, 제품명, 경쟁 제품명, 카테고리, 조사 기간, 제외 키워드를 입력한다.
- 예: `브랜드A 블루 수딩 앰플`, `민감성 앰플`, `배송`, `피부 자극`, `환불`, `냄새`, `펌프 불량`

### 4-2. 자동 검색어 확장

- 제품명에서 공식명, 줄임말, 색상 표현, 패키지 표현, 신상/구버전 표현, 오타 후보를 생성한다.
- 부정 VOC 탐색을 위해 `단점`, `후기`, `불만`, `환불`, `고장`, `자극`, `별로`, `실망`, `배송`, `깨짐` 같은 이슈 키워드를 조합한다.

### 4-3. 공개 채널 시장조사 수집

- 1차 수집은 공식 API 또는 공개 검색 API를 우선 사용한다.
- 네이버 검색 API: 블로그, 뉴스, 카페글, 쇼핑 검색 결과를 JSON/XML로 조회할 수 있다.
- 네이버 DataLab API: 검색어 그룹별 트렌드 변화를 확인할 수 있다.
- YouTube Data API: 공개 영상 댓글을 `commentThreads.list`로 조회할 수 있다.
- Google Play Reviews API: 앱 소유자가 권한을 가진 앱 리뷰를 조회할 수 있으므로 일반 경쟁사 시장조사용이 아니라 권한 있는 앱 리뷰 분석에만 사용한다.
- `insane-search`: 공식 API로 본문 확보가 어려운 공개 페이지에 대해 공개 API, 피드, 모바일 URL, RSS, 공개 JSON, 헤드리스 브라우저 순서로 읽기를 시도하는 보조 수집기로 사용한다.
- 보유 VOC/CSV 업로드는 메인 수집 방식이 아니라, 브랜드가 이미 가진 CS 로그·리뷰 덤프·상담 메모를 자동 시장조사 결과와 합쳐 비교하는 보강 입력으로 유지한다.

### 4-4. 출처/권한 필터링

- 로그인, 페이월, 비공개 커뮤니티, 인증이 필요한 페이지는 `권한 필요` 또는 `고객 제공 필요` 상태로 라우팅한다.
- 자동 접근 가능한 공식 API/공개 페이지는 즉시 분석하고, 권한이 필요한 데이터는 브랜드가 제공한 CSV, 내부 VOC 커넥터, 제휴 API로 보강한다.
- 각 문서마다 `source_url`, `source_type`, `collection_method`, `retrieved_at`, `permission_status`, `text_scope`, `next_action`을 저장한다.
- 본문 전체 저장이 위험한 채널은 원문 재게시를 하지 않고 URL, 제목, 짧은 발췌, 분석 결과, 근거 위치만 저장한다.

### 4-5. 개인정보/민감정보 마스킹

- 이름, 전화번호, 이메일, 주문번호, 주소, 계정 ID처럼 식별 가능성이 있는 정보를 분석 전에 마스킹한다.
- 마스킹된 원문만 LLM 분석 단계로 넘긴다.

### 4-6. 제품명 정규화

- 원문에 나온 `파란 앰플`, `블루앰플`, `이번 신상`, `그 펌프 선크림` 같은 표현을 제품 마스터 후보와 매칭한다.
- 확신도가 낮으면 자동 확정하지 않고 `검토 필요`로 표시한다.

### 4-7. AI 감성·비꼼·불만 유형 분류

- LLM을 사용해 `표면 감성`, `의도 감성`, `비꼼 여부`, `불만 유형`, `근거 문장`, `확신도`를 구조화한다.
- 자유 텍스트가 아니라 JSON Schema 기반 구조화 출력으로 결과를 받는다.

### 4-8. 이슈 클러스터링 및 대시보드 시각화

- 비슷한 불만을 “배송 지연”, “피부 자극”, “펌프 불량”, “냄새 불만”, “파손 배송” 같은 반복 이슈 단위로 묶는다.
- 채널별 언급량, 날짜별 증가 추이, 주요 원문 근거, 대표 키워드를 시각화한다.

### 4-9. AI 개선 퀘스트 자동 생성

- 반복 이슈를 단순 리포트가 아니라 담당자가 실행할 수 있는 개선 과제로 바꾼다.
- 퀘스트는 `이슈`, `근거`, `원인 가설`, `담당 부서`, `실행 액션`, `우선순위`, `검토 상태`를 포함한다.

### 4-10. 주간 시장 VOC 리포트 생성

- 주요 급증 이슈, 신규 불만 유형, 경쟁 제품 대비 자주 언급되는 단점, 완료/미처리 퀘스트를 요약한다.
- PDF 또는 노션 복붙용 Markdown 리포트로 내보낸다.

## 5. 서비스 흐름

### 5-1. 1~2분 발표 흐름

1. 담당자가 브랜드명과 제품명, 경쟁 제품명을 입력한다.
2. 시스템이 제품명 별칭과 불만 키워드를 자동으로 확장한다.
3. 네이버 검색 API, YouTube Data API, 공개 페이지 리더(insane-search 활용 가능)를 통해 공개 시장 반응을 수집한다.
4. 로그인/페이월/비공개 페이지는 `권한 필요` 상태로 분기하고, 고객 제공 데이터나 제휴 API로 보강할 수 있게 출처와 다음 액션을 기록한다.
5. 보유 VOC/CSV가 있으면 같은 스키마로 합류시켜 공개 VOC와 비교한다.
6. 수집된 텍스트에서 개인정보를 마스킹하고 제품명 별칭을 공식 제품명으로 정규화한다.
7. AI가 비꼼/반어까지 고려해 불만 유형을 분류하고 반복 이슈를 묶는다.
8. 반복 이슈를 담당자에게 할당 가능한 개선 퀘스트로 전환한다.

### 6-1. 검증할 챌린징 포인트

| 합법적 수집 경계 | 같은 수집 기능을 유지하면서 권한별로 어떤 처리 경로를 태울지 확인 | 공식 API/공개 페이지는 자동 수집, 로그인·페이월·비공개는 권한 필요 상태로 표시 후 고객 제공 CSV/내부 커넥터/제휴 API로 보강 |

| 제품명 정규화 | “그 파란 앰플”, “이번 신상”처럼 모호한 표현을 공식 제품명 후보로 연결할 수 있는지 확인 | alias table + 후보 점수 + 검토 필요 상태 |
| 비꼼/반어 감지 | 표면상 긍정처럼 보이는 부정 VOC를 놓치지 않는지 확인 | 표면 감성/의도 감성/비꼼 신호/근거 문장 분리 |
| 노이즈 처리 | 광고성 글, 무관한 글, 복붙 글, 중복 글을 어떻게 낮은 가중치로 처리할지 확인 | source score, duplicate hash, 광고/협찬 키워드 플래그 |
| 개인정보 마스킹 | 공개 댓글과 글에 포함된 전화번호, 이름, 주문번호를 분석 전에 처리할 수 있는지 확인 | PII regex + LLM 보조 마스킹 + 원문 저장 최소화 |
| 대표성 한계 | 공개 데이터만으로 시장 전체를 대표한다고 말할 수 없는 문제 | “시장 전체”가 아니라 “공개 채널 조기 신호 탐지”로 포지셔닝하고 내부 VOC와 분리 표기 |
| 퀘스트 품질 | 분석 결과가 실행 가능한 업무 카드가 되는지 확인 | 이슈 → 근거 → 원인 가설 → 담당 부서 → 실행 액션 → 우선순위 필드 강제 |

## 7. 차별점 / 어필 포인트

- CSV 업로드만 요구하는 도구가 아니라, 브랜드/제품명만 넣어 공개 시장 반응을 자동 조사하고 CSV/사내 VOC는 보강 데이터로 합치는 파이프라인
- 단순 감성 분석이 아니라 `수집 → 출처 검증 → 정제 → 제품명 정규화 → 비꼼 감지 → 이슈 클러스터링 → 퀘스트 생성`으로 이어지는 end-to-end 구조
- 무단 크롤링이 아니라 공식 API와 공개 페이지 리더를 조합하고, 권한이 필요한 데이터는 고객 제공/제휴 연동으로 라우팅하는 안전한 수집 설계
- 제품명 별칭과 비꼼/반어처럼 실제 VOC 분석에서 어려운 지점을 PoC 전면에 배치
- 브랜드, CS, PM, 마케팅 대행사가 사용할 수 있는 B2B SaaS 확장 가능성

---

## 8. 기술 챌린지 / 기술적 가능성

- **검색어 확장 엔진:** 브랜드명/제품명/경쟁 제품명에서 별칭, 오타, 색상 표현, 부정 VOC 키워드를 자동 생성한다.
- **멀티소스 수집 오케스트레이션:** 네이버 검색 API, 네이버 DataLab, YouTube Data API, insane-search 기반 공개 페이지 리더, 고객 제공 CSV를 수집 단계별로 연결한다.
- **수집 권한 게이트:** 수집 전후로 로그인/페이월/비공개/약관 위험 페이지를 권한 상태별로 분기하고, 자동 수집·고객 제공·제휴 API·분석 보류 중 하나의 다음 액션을 기록한다.
- **본문 추출 실패 대응:** 공식 API에서 URL/요약만 확보한 경우 insane-search로 공개 본문 읽기를 시도하고, 실패 시 제목/요약 수준 분석으로 낮은 신뢰도를 부여한다.
- **제품명 정규화:** 원문 표현을 제품 마스터 후보와 연결하되, 확신도가 낮으면 자동 확정하지 않는다.
- **비꼼/반어 분류:** 표면 감성과 실제 의도 감성을 분리해 부정 VOC를 놓치지 않도록 한다.
- **이슈-퀘스트 변환:** 반복 이슈를 담당자, 우선순위, 액션이 있는 업무 카드로 변환한다.

| 비꼼/반어 오분류 | 표면 감성/의도 감성/비꼼 신호/근거 문장 분리 |
| 개인정보 포함 가능성 | 수집 직후 PII 마스킹 후 분석, 원문 저장 최소화 |
| 저작권/본문 저장 리스크 | URL, 제목, 짧은 발췌, 분석 결과 중심 저장. 장문 원문 재게시 금지 |
| 실행 불가능한 퀘스트 생성 | 퀘스트 스키마에 담당 부서, 액션, 우선순위, 근거 문장 필수화 |

## 12. 유사 서비스 / 참고 자료

- 네이버 검색 API: 블로그, 뉴스, 쇼핑, 카페글 등 네이버 검색 결과를 분야별로 조회할 수 있는 공식 API
- 네이버 DataLab API: 검색어 그룹별 트렌드 변화를 확인할 수 있는 공식 API
- YouTube Data API: 공개 영상 댓글을 조회할 수 있는 공식 API
- Google Play Reviews API: 권한 있는 앱 리뷰 조회 API. 경쟁사 일반 시장조사용이 아니라 소유 앱/권한 있는 앱 분석용
- insane-search: 공개 페이지·공개 API·피드·아카이브·브라우저 공개 응답으로 접근 가능한 텍스트를 읽는 공개 페이지 리더
- OpenAI Structured Outputs: VOC 분석 결과를 JSON Schema 기반으로 고정 형식 출력

## 13. 멘토 피드백 / 메모

- 수집이 관건 → CSV 업로드 기능은 보강 입력으로 유지하고, 기본 가치는 자동 시장조사 파이프라인으로 확장
- 핵심 PoC는 “분석 모델 성능”보다 “공개 VOC를 안전하게 수집하고, 별칭/비꼼/노이즈를 처리해 업무 퀘스트로 바꾸는 과정”
- insane-search는 수집 파이프라인의 보조 도구로 활용 가능하지만, 인증 우회/비공개 수집처럼 보이면 안 됨
